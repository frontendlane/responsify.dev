Responsify

Responsify generates a CSS calc() value that gradually changes an element's width, margin, padding, etc. between two viewport sizes. Unlike % or vw, generated calc() value can change at nonproportional and negative rate. A common use case is to transform adaptive design based on @media queries into fluid design.

How it works

Let's say you have an image that needs to be 100% on resolutions below 1000px and 30% above that. Because you use mobile first approach, you start with:

img {
    width: 100%;
}

and for desktop you add:

@media (min-width: 1000px) {
    img {
        width: 30%;
    }
}

You want to gradually transition between these two values so you reserve the 900px - 1000px range to smooth this out. You add another @media query with a magic 🪄 calc() value generated from the form below:

@media (min-width: 900px) {
    img {
        width: calc(6300px - 600%);
    }
}

which you place before the first @media query and you're done! 🎉 You get something that behaves like the <div> element at the top of the page (resize the browser to see it in action).

CSS comment and rounding

Tweaking generated value later is hard because input values aren't preserved. So it's a good practice to add a comment documenting them.

When you generate a calc() value with the form above, the CSS comment is added to your clipboard as well, but you can turn this off. Should you decide to use your own comment format, make sure to specify the units as well. Below is an example of what would be copied to your clipboard.

width: calc(9.286px + 4.911vw); /* Responsified - viewport lower bound: 320px; viewport upper bound: 1440px; element lower bound: 25px; element upper bound: 80px; */

I added Responsified at the start of the comment because I found it useful to have a unique string by which I can easily find all "responsified" values.

calc() values are rounded to 3 decimal places because adding more decimals doesn't affect dimensions of rendered elements, even when dealing with large numbers. Unfortunately, I can't find the article where I read this to back it up. 😕

Sass @function

Sass @function

You can use Responsify as a Sass @function. First, download _responsify.scss where your root Sass file is (usually called style.scss so look for that). Then unzip the downloaded file and finally, at the top of your root Sass file add @import "responsify"; (if you're using Dart Sass, add the recommended @use "responsify"; instead).

The added bonus of using the Sass @function is that you don't need comments documenting input values. 🙌

Note: At the moment, Sass @function doesn't support ch unit.

Sass usage examples

Example 1 - featuring px

Example 1 - featuring px

Responsify <div>'s padding from 40px at 320px viewport width to 90px at 1024px viewport width. Note that calc() will be applied at all viewport widths, not only between 320px and 1024px.
div {
    padding: responsify(320, 1024, 40, 90, "px");
}
CSS output
div {
    padding: calc(-2232.72727px + 7.10227vw);
}

Example 2 - featuring %

Example 2 - featuring %

Responsify <img>'s height from 60% at 320px viewport width to 30% at 1024px viewport width. Same as in the previous example, the calc() will be applied at all viewport widths.
img {
    height: responsify(320, 1024, 60, 30, "%");
}
CSS output
img {
    height: calc(-5044.36364px + 16.36364%);
}

Example 3 - featuring @media query, optional $unit argument

Example 3 - featuring @media query, optional $unit argument

Responsify <section>'s margin-top from 50px at 320px viewport width to 100px at 1000px viewport width. Due to s @media query in this example, the calc() will be applied up to and including 1000px viewport width. Note the omitted optional "px" argument at the end.
@media (max-width: 1000px) {
    section {
        margin-top: responsify(320, 1000, 50, 100);
    }
}
CSS output
@media (max-width: 1000px) {
    section {
        margin-top: calc(-2302.94118px + 7.35294vw);
    }
}

Example 4 - featuring vw, clamp()

Example 4 - featuring vw, clamp()

Responsify <video>'s width from 100vw at 320px viewport width to to 60vw at 1024px viewport width. In this example width is lower bound at 300px and upper bound at 1800px via clamp() function.
video {
    width: clamp(300px, responsify(320, 1024, 100, 60, "vw"), 1800px);
}
CSS output
video {
    width: clamp(300px, calc(-13061.81818px + 41.81818vw), 1800px);
}

Why I built this

tl;dr: much lazy. such automation.

I made Responsify to scratch my own itch. 💆‍♂️ Working at a digital agency, I usually get design files with 3 breakpoints (sometimes 2, sometimes only 1, and sometimes none(!) but that's another story) with the expectation that when implemented, the content will reflow, adapt, and still look good in those in-between sizes. Pretty standard stuff.

But how should I handle element sizes and spacings in between those breakpoints? Should I

  1. use the pixel values from design files without worrying if it'll look disproportionate at certain viewport sizes? 🤨
  2. try to convert pixel values to % or vw so I get a more fluid design, but run the risk of ballooning these sizes as viewport size goes up, especially if the breakpoints are too far apart (e.g. only two breakpoints)? 🤨
  3. come up with my own way to exactly match the design at the delivered breakpoints AND gradually transition the value between those breakpoints? 🤔

Guess which one I picked? 😬 My first approach was to calculate the line equation between two breakpoints on paper (college math and stats classes came in real handy here 🧠) and then just plug that into a CSS calc() function. Quickly I realized I can streamline and automate the process so here's Responsify! 🤖
